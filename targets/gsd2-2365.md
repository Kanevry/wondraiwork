## Problem
When GSD2 tears down a git worktree after completing a task, it can destroy uncommitted data in the worktree directory. If the agent's work is interrupted or fails partway through, the teardown process does not check for unsaved changes before removing the worktree, leading to irreversible data loss.

## Impact
Affects GSD2 users who rely on worktree-based task isolation. Data loss bugs are high severity regardless of user count. The issue is 15 days old, indicating it is a recently discovered regression or gap.

## Approach
1. Locate the worktree teardown logic (likely in a cleanup/lifecycle module)
2. Add a pre-teardown check: `git diff --quiet` and `git diff --cached --quiet` to detect uncommitted changes
3. If uncommitted changes exist: either stash them, commit to a recovery branch, or abort teardown with a warning
4. Add tests for the teardown-with-dirty-worktree scenario

## Notes
This is a safety-critical fix. The approach should err on the side of preserving data -- if there is any doubt, do not delete. A stash-and-warn approach is likely the safest default. The fix is relatively small but has high impact for trustworthiness of the tool.
